# Verifying an Avark release

Avark is a self-custody Bitcoin wallet. Before you install a build that will hold your sats, you should verify that:

1. The binary was produced by Avark's CI from a commit you can inspect.
2. It has not been modified in transit or tampered with on the mirror that served it.

Every official release ships with two extra files that make this possible:

- `SHA256SUMS` — SHA-256 digests of every artifact in the release.
- `SHA256SUMS.asc` — a detached GPG signature of `SHA256SUMS`, made by the Avark release key.

If either file is missing, **do not install the release**. Open an issue instead.

## One-time setup: import the signing key

The public key lives in this repo at [`docs/release-signing-key.asc`](release-signing-key.asc). Import it once:

```bash
gpg --import docs/release-signing-key.asc
```

Then verify that the fingerprint you imported matches the one published here:

```
F9DD ABD2 E4A9 0063 99AF  A5B4 3A00 9E0D 4F6D 73BE
```

If the fingerprint does not match, **stop**. Something is wrong — either this README is out of date, the key file has been tampered with, or you imported a different key.

## Per-release verification

From the directory where you downloaded the release artifacts:

```bash
# Verify the checksum file's signature was made by the Avark release key.
gpg --verify SHA256SUMS.asc SHA256SUMS
```

Expected output ends with: `Good signature from "Avark Releases <…>"`. A `WARNING: This key is not certified with a trusted signature` line is normal — you haven't signed the key yourself. The important word is **Good**.

Then check each file's digest:

```bash
sha256sum -c SHA256SUMS
```

Every line should end in `: OK`. If any line says `FAILED`, that artifact was corrupted or swapped — do not install it.

## What's signed

- `avark-<version>-android-arm64-v8a.apk` — modern 64-bit phones (most people).
- `avark-<version>-android-armeabi-v7a.apk` — older / budget 32-bit phones.

iOS builds (`.ipa`) are not yet published from CI — they'll return to the release set once team signing is sorted out. Google Play AAB is also not published here; Play uses a different signing flow (Play App Signing) that would need its own workflow.

Both APKs are signed by the Android release keystore. Inspect a signature with:

```bash
apksigner verify --print-certs <downloaded-file>.apk
```

## Which APK should I download?

**If you don't know:** pick **`arm64-v8a`**. That's essentially every Android phone made from ~2017 onward.

**If `arm64-v8a` refuses to install** with an error like `INSTALL_FAILED_NO_MATCHING_ABIS`, your phone is 32-bit — download `armeabi-v7a` instead. This is typical for:

- Budget Android phones sold in emerging markets (many Tecno, Infinix, itel, and older Samsung A0x Core models).
- Phones from ~2015 or earlier.
- Devices running Android Go on low-end chipsets (MediaTek MT6737 / MT6739, Unisoc SC9832, etc.).

**How to check before downloading:**

```bash
# On a Mac/Linux with the phone plugged in via USB and adb-authorized:
adb shell getprop ro.product.cpu.abi
```

`arm64-v8a` → download the arm64-v8a APK. `armeabi-v7a` → download the armeabi-v7a APK. Anything else (`x86`, `x86_64`) means you're on an emulator; either ABI will work via translation, but arm64-v8a is faster on modern emulators.

## What's not signed

Source tarballs generated by GitHub (`Source code (zip)`, `Source code (tar.gz)`) are not included in `SHA256SUMS`. GitHub produces those server-side after the release is published; they are outside the release-signing pipeline. Clone the tagged commit from git if you want source.

## Reporting problems

If `gpg --verify` or `sha256sum -c` ever fails on an official release, **do not dismiss it**. Open an issue (or email the maintainer privately) before installing anything else from that release.
